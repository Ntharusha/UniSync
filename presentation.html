Introduction

Project Background

🏫 Context

In modern universities, communication between students and lecturers regarding academic queries, project discussions, and administrative approvals is highly unstructured.

Common tools like email or WhatsApp lead to delayed replies, schedule conflicts, and a lack of transparency for both parties.

🎯 The Solution: UniSync

UniSync replaces informal messaging with a real-time, timetable-aware booking workspace where students can seamlessly schedule verified, non-conflicting time slots.

Ensures optimal coordination by integrating official timetables and academic calendars into a single, unified channel.

Requirement Analysis (1/2)

Functional Requirements

👨‍🎓 Student Role

  • Browse lecturer profiles and view true availability schedules
  • Select free slots & specify request title, reason & priority
  • Real-time notifications upon request status updates
  • Direct message communication inside the appointment context

👩‍🏫 Lecturer Role

  • Define default weekly office hours and blackout times
  • Upload official timetables to automatically eliminate slots
  • Approve, Reject, or reschedule student requests
  • Real-time chat interactions with students

🛡️ Administrator Role

  • Manage all student, lecturer, and admin accounts
  • Execute bulk CSV/Excel user imports
  • Monitor and review system-wide audit logs
  • Maintain global timetable data & department details
Requirement Analysis (2/2)

System Qualities & Non-Functional

🛡️ Security & Privacy

Password hashing via bcryptjs and route authorization using JSON Web Tokens (JWT).

Explicit role validation middleware prevents cross-role privilege escalation (e.g. students accessing admin screens).

⚡ Real-Time Responsiveness

Instant booking state changes, incoming chat alerts, and urgent reminders delivered under 500ms latency using persistent **WebSocket (Socket.io)** connections.

📈 Scalability & Reliability

Timetable parsing queue management using **BullMQ** with a Redis-backed engine to guarantee background job reliability without blocking server CPU processes.

🔧 Database Integrity

Comprehensive Mongoose schemas with indexing on keys (e.g., email, department, date) to minimize lookup latency and guarantee absolute data consistency.

System Design (1/3)

Architecture & Technology Stack

🛠️ Technology Stack

Web Platform:

Mobile Platform:

React Native Expo Zustand TanStack Query

📡 System Architecture

A decoupled Client-Server architecture utilizing:

  • REST API Layer for general transactional processes (auth, configuration, user data).
  • WebSocket Layer for bidirectionally pushed updates (chat messages, real-time slot bookings).
  • Redis Queue Engine for asynchronous processing (parsing massive files, scheduling notification deliveries).
System Design (2/3)

Database Schema & Data Models

👤 User Schema

Maintains registration details, roles, departments, and specific academic metadata.

  • Name, Email, Password
  • Role (Student / Lecturer / Admin)
  • Department, Faculty
  • Registration No / Employee ID

📅 Appointment Schema

Models transactions between student requestors and host lecturers.

  • Student Ref (ObjectId)
  • Lecturer Ref (ObjectId)
  • Time Block Details
  • Priority (Normal/Urgent/Emergency)
  • Status (Pending/Approved/Rejected)

🗓️ Timetable Block Schema

Ensures correct booking checks by mapping scheduled lectures.

  • Lecturer Ref
  • Day of Week
  • Start & End Times
  • Course Details
  • Ingestion Source File
System Design (3/3)

Timetable Conflict & Priority Rules

🔄 Conflict Detection Workflow

1

Read Base Availability

Retrieves the lecturer's customized standard working hour windows.

2

Subtract Lecture Times

Removes any times matching the lecturer's parsed academic timetable classes.

3

Filter Existing Bookings

Eliminates slots already occupied by active approved appointments.

🔥 Priority Override Rules

To ensure critical student situations are resolved, the system operates on a priority ranking:

Emergency (High) Urgent (Medium) Normal (Low)

If a request is flagged as an **Emergency** and validated by a lecturer, the slot engine can systematically prioritize the scheduling over any standard low-priority conflict.
